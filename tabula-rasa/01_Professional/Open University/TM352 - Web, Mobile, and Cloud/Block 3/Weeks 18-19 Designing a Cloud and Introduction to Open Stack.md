# Week 18-19 - Designing a Cloud and OpenStack:

# 1. Introduction:

Part 1 introduced the conceptual cloud — definitions, features, characteristics, layered models, and the enabling technologies of virtualisation and scaling. This part turns to concrete examples, looking in greater detail at what goes into designing a small cloud.

We start by exploring a traditional deployment of equipment for a small to medium web application. Then we describe the equivalent set-up in a cloud environment, such as could be built using Amazon's Web Services or an in-house solution using OpenStack.

Almost everything in the traditional deployment has its counterpart in the cloud. The difference is that in the cloud, most of the infrastructure is created dynamically as and when requested by a tenant. The underlying physical networks, servers and disks must exist, but the dynamic element is how a portion of those resources are allocated to a single tenant.

**Note:** The descriptions assume familiarity with IP addressing, subnetting and CIDR prefix/length notation. See Appendix 1 if these are new concepts.

# 3. Traditional Infrastructure Design:

The infrastructure comprises the key elements for a small web application. To keep things simple there is no redundancy, so if something fails the whole application becomes inaccessible. The goal is to focus on each element's role so you can see how they map into the cloud.

The web application is a simple online shop employing a web server front-end and a database back-end. Scaling-out shares the workload between three web servers. The application uses a stateless design so any web server can respond to any request, with essential user data held in a cookie.

## 3.1. Network Time Service (NTP):

The **global NTP server** sits outside the application but provides crucial data to synchronise all server clocks using the network time protocol (NTP).

The **local NTP server** acts as a secondary synchronisation source — it derives its clock data from the global NTP server and synchronises all other servers in the application. Using a secondary source reduces internet traffic and can improve security.

**Why time synchronisation matters:**

- When data generated by one server is stored on another (e.g. customer orders from web server to database server), synchronised clocks are essential
- Web browsers check if content has changed since the previous download by including timestamps in HTTP requests — if file timestamps differ by more than a few seconds, content is sent again unnecessarily

## 3.2. Domain Name Service (DNS):

The **global DNS server** converts fully qualified domain names (e.g. <www.open.ac.uk>) into IP addresses (e.g. 137.108.198.32). Messages sent over the internet use IP addresses, not friendly names.

**Note:** In Block 1, DNS referred to the domain name system (the entire distributed system). Here DNS refers to individual servers providing a service to applications.

### How DNS Works:

DNS relies on a hierarchical, distributed database of records about domains. At the highest level are domains like .UK, .COM and .TV, responsible for records of sub-domains (.CO.UK, .AC.UK). Sub-domain records point to another DNS server holding information about its own sub-domains.

**Example DNS query flow (visiting <www.open.ac.uk> for the first time):**

1. Your computer sends a query to the DNS server assigned to your network interface (e.g. your wireless router at 192.168.0.1)
2. The router can't answer, so sends its own query to its designated DNS server (your ISP's DNS servers)
3. The ISP continues the query to the top-level DNS server for .UK
4. Queries go down the hierarchy: .UK → .AC.UK → .OPEN.AC.UK
5. The OPEN.AC.UK DNS server knows the IP address of <www.open.ac.uk> and returns it back up through the hierarchy
6. Your computer uses the IP address to send an HTTP request

In practice, one server takes on the role of **"resolver"** (normally your ISP). The resolver takes responsibility for resolving the domain address to an IP address, querying down the hierarchy and returning the result directly to the originating machine.

**The local DNS server** also handles queries for IP addresses across the local network — web servers need to connect to the NTP server, the load balancer needs to connect to web servers, etc.

To reduce DNS queries, each computer stores successful query results in a **DNS cache** and refers to it before starting a new query. Alternatively, IP addresses and names can be stored in a local **hosts file**.

## 3.3. Router/NAT:

### Router:

The router fulfils two essential functions:

1. **Signal conversion:** Converting low-level signals and packets between network types — e.g. IEEE 802.3 (Ethernet) for wired networks and ADSL for internet connection
2. **Network interconnection:** Interconnecting different networks (between organisations or departmental networks). In this role it is called a **gateway**

In the gateway role, the router is assigned an IP address for each connected network. The router makes routing decisions based on an internal **routing table** built over time as traffic flows through its ports (some routers allow manual configuration).

**Example home router port details:**

|Setting|Broadband Port|LAN Port|
|---|---|---|
|MAC Address|00:23:48:f3:0a:0d|00:23:48:f3:0a:0b|
|IPv4 Address|176.249.64.184|192.168.0.1|
|IPv4 Subnet Mask|255.255.255.255|255.255.255.0|
|Gateway IPv4 Address|2.127.238.94|—|
|IPv4 DNS Server|90.207.238.97, 90.207.238.99|—|

The **gateway IPv4 address** refers to a device assumed to know how to forward IP packets to other networks — in this case, a router at the ISP.

When connecting a computer to the router, the computer's network interface is assigned an IPv4 address, subnet mask, DNS address and default gateway address.

**Key difference in cloud vs traditional:** In a traditional network the router primarily handles traffic to/from the internet, but in the cloud the routing function is essential for passing packets between subnets.

### NAT (Network Address Translation):

Your home router handles NAT to enable internet traffic exchange with home devices. Without NAT, every device would require its own public IP address, and there are insufficient addresses available (until IPv6 is fully deployed).

**Private address blocks** (RFC 1918, set aside by the IETF):

|Range|Number of addresses|CIDR|
|---|---|---|
|10.0.0.0 – 10.255.255.255|16,777,216|10.0.0.0/8|
|172.16.0.0 – 172.31.255.255|1,048,576|172.16.0.0/12|
|192.168.0.0 – 192.168.255.255|65,536|192.168.0.0/16|

Anyone can use these for their networks, but they should not be connected directly to the internet.

**How NAT works:**

1. Your computer (192.168.0.2) sends an HTTP request to an internet destination
2. The computer detects the destination is not on the local network and directs the packet to the default gateway (192.168.0.1)
3. The router detects the source address is private, so it replaces the local IP with the router's internet IP address
4. The router saves the local source and remote destination in a table
5. When a response arrives, the router looks up the original local address and forwards the response

**IP addresses during packet exchange via NAT:**

|Packet Exchange|Source IP|Destination IP|
|---|---|---|
|Computer to router|192.168.0.2|137.108.64.204|
|Router to website|176.249.64.184|137.108.64.204|
|Website to router|137.108.64.204|176.249.64.184|
|Router to computer|137.108.64.204|192.168.0.2|

To handle different connections from different devices to the same destination, the router adds a unique **source port number** to its own IP address to uniquely identify each TCP request.

## 3.4. Firewall/DMZ:

### Firewall:

The firewall blocks unwanted packets travelling between the internet and the LAN to protect servers behind it.

**Three generations of firewalls:**

1. **Packet filtering (1st generation):** Examines IP address and port number — packets only pass if both are approved. Does not check content
2. **Stateful filtering (2nd generation):** Extends filtering to include connection state — whether the packet is for a new connection, existing connection, or neither. Only new and existing connections pass
3. **Application layer filtering (3rd generation):** Operates at the application layer, investigating message content for protocols like HTTP, FTP, SMTP, DNS. Non-conforming messages or those with inappropriate data are discarded

Many modern firewalls can also function as termination points for SSL/TLS and VPN connections, off-loading work from servers.

### DMZ (Demilitarised Zone):

A single firewall is a single point of failure. The DMZ architecture provides multiple levels of protection using **two firewalls**:

- **Exterior firewall:** First line of defence — permits web server access for all internet users, limits email server access to authenticated users
- **Interior firewall:** Blocks all unsolicited internet traffic from reaching the intranet, and blocks traffic from DMZ servers (which may be compromised) from reaching the intranet

The region between the two firewalls is the **demilitarised zone**, where externally accessible services are located (web servers, email servers).

A connection between the database server and web server can be achieved by opening specific ports in the interior firewall, though additional security measures should be considered.

## 3.5. Other Elements:

**Load balancer:** Can take the form of dedicated hardware or software running on another server.

**Servers:** Web servers in a scale-out configuration (could be configured as a high-availability cluster). Each server requires multiple network interface cards (NICs): one for the load balancer, one for the database via the internal firewall, and potentially a third for a private management network for software updates and logging.

**Database:** Works with web servers to store user credentials, purchase orders and other persistent data. Protected from web servers by an internal firewall to restrict IP addresses and ports in case a server is compromised.

**Other options:** Static pages on caching servers, dedicated authentication servers, dedicated encryption hardware or load balancer handling SSL.

## 3.6. Key Points:

The purpose of this traditional network architecture example is to illustrate how many pieces of equipment are involved in creating a simple website. Much of the functionality can be provided as software rather than hardware (e.g. your home router provides DNS through software). This ability to use software modules is key to cloud computing success.

Whether hardware or software, each item requires setting up, configuring, updating, patching and constant monitoring. The administrative overhead is considerable. A hosting company may provide all equipment as a package, but it can still take several days to set up — hardly the on-demand, self-service and elasticity promised by the cloud.

# 4. OpenStack Cloud Architecture:

This section explores how the simple web application might be built in a cloud environment using the OpenStack cloud computing project.

**Two main differences between OpenStack and AWS:**

1. **OpenStack** is open-source (access to source code), whereas **AWS** is proprietary (use but not download or change)
2. **AWS** is an outsourcing solution running on AWS-owned machines (hybrid configurations possible). **OpenStack** can be downloaded and run on your own machines, though outsourcing providers and support companies also exist (the OU's platform is a supported RedHat variant)

Both options provide for autoscaling of VM instances based on rules defined by the tenant (based on network traffic or a timed schedule). The cloud utilises the same standards and protocols as traditional networks — DNS, NTP, routing, firewalls, etc. are all required.

OpenStack is an open-source IaaS project. The OpenStack Community Welcome Guide describes it as a global collaboration of developers and cloud computing technologists producing the open standard cloud computing platform for both public and private clouds.

### Cloud Architecture Overview:

The architecture has two halves. The **top half** contains the same equipment as the traditional network (DNS servers, NTP servers, router, firewall). The **bottom half** represents OpenStack cloud elements and the tenant website.

Most top-half equipment performs the same function as in the traditional network. The firewall must be configured to open extra ports for cloud configuration and management traffic. The local DNS server handles records for new websites created dynamically in the cloud.

The bottom half contains **five key blocks**: three represent computer hardware (**Controller**, **Network**, **Compute**) and two represent disk storage (**Images**, **Storage**).

The hardware is interconnected by **three separate networks**:

1. **Management network:** Handles management traffic and data between computers
2. **VM traffic network:** Handles network traffic to and from the website
3. **Internet network:** Carries internet traffic exchanged with the cloud

## 4.1. Controller Node:

The controller node manages the cloud:

- Schedules new VMs to start and stop
- Manages images used to create VMs
- Handles authentication for cloud administrators and tenants
- Gathers detailed information about cloud usage

This is achieved by running **services** (programs launched at start-up that run in the background). Services communicate via a **message queue** — one service posts a message and another reads it later, meaning services don't have to wait for responses.

The controller node decides where a VM will be created based on the **flavor** chosen and the **resources** (processor, memory) available on compute nodes.

## 4.2. Network Node:

The network node manages network elements used by VMs to communicate with the internet:

- Assigns IP addresses to VMs
- Routes traffic among VMs and between VMs and the internet

The solution uses services that mimic the internetwork layer and data link layer of the OSI protocol stack.

**When a VM is launched:**

1. The controller node sends a message to the network node to create required networks
2. The network node assigns IP addresses using VM network interface information and sets up routes
3. The first IP address assigned is for a **private network** linking the VM to the network node (allowing cloud nodes to communicate with the VM)
4. A **security group** is created (equivalent of a firewall) containing details of accessible ports and permitted user accounts
5. If the VM needs internet access, it gets a **floating IP** address used solely for internet traffic, plus necessary routes

The network node has optional services providing **load balancing** and **firewall protection**.

## 4.3. Compute Node:

The compute node is the host computer for guest VMs. When a new VM launches, the controller posts a message and the compute node loads the image from image storage.

The compute node also:

- Runs a network service configuring interfaces and bridges for VM communication (since all VM network traffic must use the host computer's network interfaces)
- Uses **software-defined route tables** instead of hardware routers to define addresses for data exchange between collaborating VMs and internet gateways
- Allocates **ephemeral storage** by default (sufficient for boot-up and temporary data — discarded on shutdown). **Persistent storage** requires creating a **volume** linked to the VM at launch time
- Gathers information about VM activities for tenant billing, passed to the controller via message queue

## 4.4. Images:

Disk storage for VM images, managed by a controller node service. New images can be loaded, removed, shared between tenants, or marked as private for a single tenant.

## 4.5. Storage:

Disk storage for persistent data required by VMs, controlled by a controller node service. Takes the form of a virtual disk drive attached to an individual VM — could store static web pages, images, dynamic pages (PHP scripts) and web server logs.

## 4.6. Management:

OpenStack has two modes of control:

1. **Web-based interface** (Horizon dashboard)
2. **Application programming interfaces (APIs)** for RESTful web services

Using these tools, a tenant can launch/review instances, create/edit flavors, load/unload images, create storage volumes and monitor usage.

## 4.7. A Website in OpenStack:

A website can be created entirely through the OpenStack management interface using controller, network and compute node services. No further hardware is required — everything is achieved through software.

**Setup process:**

1. Create and save a web server image (e.g. Linux + Apache) and a database server image (e.g. MySQL) to the images node
2. Allocate persistent storage for the database via a volume on the storage node
3. Plan database design carefully as it will be shared between multiple web servers
4. Launch the web server VM connected to the internet with a private network to the database server
5. The controller monitors the website VM and can launch additional instances if needed
6. A load balancer service distributes requests between available servers

## 4.8. Further Options:

The simple cloud example has too many single points of failure for a commercial offering (one control node, one network node). Resilience can be improved using traditional clustering models (active/active or active/passive), multiple controller and network nodes with traditional load balancing.

There is also significant network traffic between nodes and VMs beyond the essential VM internet traffic: service messages, logging data, management data, and monitoring data.

# 5. Achieving the Cloud (NIST Characteristics in OpenStack):

**On-demand self-service:** OpenStack offers a web-based interface and APIs to manage all stages of creating, managing and monitoring a cloud application. Server, network, firewall and router exist only as software — all traditional hardware elements replaced by software counterparts. Disk storage is managed through software services like Swift, eliminating the need for hardware RAID configurations.

**Broad network access:** From an end-user perspective, deployed applications are accessible. From a tenant's perspective, network access is needed to create, manage and monitor their application.

**Resource pooling:** All nodes are shared by multiple tenants. The tenant won't know, and doesn't need to know, which node is performing which task.

**Rapid elasticity:** OpenStack provides tools to monitor workload and change deployed resources as necessary, scaling up and down in proportion with demand.

**Measured service:** OpenStack provides a rich variety of monitoring tools for tenant applications and resource measurement, viewable through the web interface or API calls.

# 6. The OpenStack Platform:

The OpenStack Foundation started in 2010 and is supported by organisations including AT&T, IBM, Red Hat, Rackspace, Yahoo, Cisco and many others. The software is written mainly in Python with JavaScript and XML, comprising over a million lines of code produced by a very large community. New releases with features and bug fixes appear every couple of months.

## 6.1. OpenStack Architecture:

OpenStack consists of an extensive set of independent services. The primary interface is the **Horizon dashboard**, which sits in front of a collection of services as a unifying interface. Horizon is configured at installation to include access only to the services being installed — if an optional service is not installed, it doesn't appear in the interface.

### Core Services:

|Service|Project Name|Description|
|---|---|---|
|Dashboard|**Horizon**|Web-based self-service portal for interacting with OpenStack services (launching instances, assigning IPs, configuring access controls)|
|Compute|**Nova**|Manages lifecycle of compute instances — spawning, scheduling and decommissioning VMs on demand. Supports large pools of VMs, horizontal scaling, and wide range of virtualisation technologies including Docker|
|Networking|**Neutron**|Provides Network-Connectivity-as-a-Service for OpenStack services. API for defining networks and attachments. Pluggable architecture supporting many networking vendors|
|Object Storage|**Swift**|Stores and retrieves unstructured data objects via RESTful HTTP-based API. Highly fault tolerant with data replication and scale-out architecture. Writes objects to multiple drives across a server cluster|
|Block Storage|**Cinder**|Provides persistent block storage to running instances. Pluggable driver architecture for block storage device management. Volumes attach to VMs, are used, then detached|
|Identity|**Keystone**|Authentication and authorisation for all OpenStack services. Provides catalogue of endpoints for all services. Checks credentials for every operation|
|Image|**Glance**|Stores and retrieves VM disk images in various back-ends. Supports formats: VHD (Hyper-V), VDI (VirtualBox), qcow2 (Qemu/KVM), VMDK (VMware). Images can be public or private|
|Telemetry|**Ceilometer**|Monitors and meters the cloud for billing, benchmarking, scalability and statistics|
|Orchestration|**Heat**|Orchestrates composite cloud applications using native HOT template format or AWS CloudFormation templates, via REST API|

**Additional services:** Sahara (Hadoop for large datasets), Trove (database system for OpenStack), Ironic (bare-metal machine deployment instead of VMs).

### Storage Types:

- **Swift (Object Storage):** Older project, stores large objects as single items. Storage and retrieval based on whole objects. Glance stores images in Swift. Distributed with redundant copies across a network for robustness and speed
- **Cinder (Block Storage):** Provides storage volumes for compute. Virtualised storage associated with VMs via Nova. A volume is an area of file system storage attached to a VM, used, then detached later

## 6.2. Accessing OpenStack:

Each service supports **four approaches to access:**

1. **Horizon dashboard:** Web-based interface (the simplest method)
2. **Command-line interface (CLI):** Requires installing OpenStack command-line clients. Commands issued from a shell or console

```bash
# List all images
glance image-list

# Show details of a specific image
glance image-show 22c7f4c4-39fc-4f1d-92ef-333cfc535eb7
```

3. **Python APIs:** Programmatic access via the Python SDK, HTTP REST clients (e.g. Linux `curl`), or browser-based REST clients

```python
import glanceclient.v2.client as glclient
glance = glclient.Client(...)
images = glance.images.list()
```

4. **YAML declarative scripting:** A YAML description creating any number of resources and actions is called a **"stack"** in OpenStack terminology

## 6.3. OpenStack at the OU:

The version available at the OU is **Xena**. Required services for a minimal installation include Nova, Neutron, Keystone and Glance, plus at least one storage service. Horizon is optional but recommended.

**Additional services installed at the OU:**

|Service|Component|Purpose|
|---|---|---|
|Orchestration|**Heat**|Specification language for defining resources and operations that OpenStack runs automatically|
|Load Balancing|**LBaaS/Octavia**|Load Balancing as a Service — supports automatic VM duplication and load balancing|
|Metering|**Ceilometer**|Monitors resource usage (e.g. load on a VM) to support orchestration decisions|

OpenStack is open source under the **Apache 2.0 licence** and free to use.

# 9. Making and Managing VMs:

## 9.1. Creating VM Images:

VM images can be created using **VirtualBox** — create a VM from an ISO source image of an operating system, install desired software (e.g. web server) while running, close down the VM, then convert the image file.

**Conversion commands (Linux):**

```bash
VBoxManage clonehd --format RAW vm.vdi vm.img
qemu-img convert -f raw vm.img -O qcow2 vm.qcow2
```

Alternatively, the VirtualBox image can be used directly in OpenStack by selecting 'vmdk' as the format. Images can also come from online archives on the OpenStack website.

## 9.2. Configuration Scripts:

When an image is launched, there can be an option to run a script immediately after the instance starts (disabled in TM352 for security reasons, but available with administrative control). The Configuration tab in Horizon's Launch Instance dialogue allows specifying a script file or typing commands directly.

The script must be written in the instance's OS scripting language or a language already installed in the image. Commands run as if typed into the console when the instance starts.

## 9.3. Two Approaches to Adding Software:

1. **Pre-built image:** Software added to the OS, saved as a new "enhanced" image for running as a VM
2. **Dynamic installation:** Software added to the basic OS dynamically when the image starts up

# Appendix: IP Addressing:

Each 32-bit IP address is split into four groups of 8 bits (an octet), each converted to a decimal number 0–255. For example: `01011010 11001111 11101110 01100001` = `90.207.238.97`.

## IP Address Structure:

IP addresses are split into two parts: a **network ID** and a **host ID**. Messages on the internet are routed between networks using the network ID, and the network identifies the device based on the host ID.

**IP address classes:**

|Class|Class Bit Pattern|Network ID Length|Host ID Length|Address Range|
|---|---|---|---|---|
|A|0|7 bits|24 bits|1.0.0.0 – 127.255.255.255|
|B|10|14 bits|16 bits|128.0.0.0 – 191.255.255.255|
|C|110|21 bits|8 bits|192.0.0.0 – 223.255.255.255|

**Network masks:**

|Class|Network Mask|
|---|---|
|A|255.0.0.0|
|B|255.255.0.0|
|C|255.255.255.0|

The number of devices per network is always two less than the maximum because all-zeros and all-ones host IDs are reserved.

## Private Address Space:

Reserved by IANA for devices not connecting directly to the internet. Also used significantly in cloud network creation:

- 10.0.0.0 – 10.255.255.255 (single class A network)
- 172.16.0.0 – 172.31.255.255 (16 contiguous class B networks)
- 192.168.0.0 – 192.168.255.255 (256 contiguous class C networks)

Routers are programmed to reject incoming packets with source addresses in private ranges.

## Subnets:

Subnetting takes some bits from the host ID to create a **subnet ID**, allowing ISPs to divide large address blocks into smaller chunks for customers, or organisations to split private address blocks into departmental networks.

**Example:** A class B network 172.24.0.0 with mask 255.255.0.0 extended by 3 bits becomes 255.255.224.0, creating 8 subnetworks each hosting 8,191 devices.

**Subnet address ranges for 172.24.0.0 with 3-bit extension:**

|Network ID|Host Range|
|---|---|
|172.24.0.0|1 – 8,190|
|172.24.32.0|8,193 – 16,382|
|172.24.64.0|16,385 – 24,574|
|172.24.96.0|24,577 – 32,766|
|172.24.128.0|32,769 – 40,958|
|172.24.160.0|40,961 – 49,150|
|172.24.192.0|49,153 – 57,342|
|172.24.224.0|57,345 – 65,534|

Do not confuse the **network mask** (used by internet routers to forward traffic) with the **subnet mask** (used within a network to locate a single host).

## CIDR Notation:

Class-based addressing has been replaced by **classless inter-domain routing (CIDR)**. The network ID and mask are combined as a **prefix/mask length** — e.g. `172.24.0.0/19` means 19 bits for the mask, leaving 13 bits for host addressing.

CIDR notation is used to specify network address space in both OpenStack and AWS. For example, the default AWS VPC address range is `10.0.0.0/16` (65,534 hosts), which can be subnetted into `10.0.0.0/24` and `10.0.1.0/24` (254 hosts each).

**Important rule:** To exchange messages between two devices, they must be part of the same subnet. The CIDR prefix/length notation can determine if two hosts are on the same subnet by applying the mask to both addresses and comparing the results.
